### 1.2 真正重要的不是“每篇都跟”，而是整理出一个故事骨架

这些论文其实在共同指向一个结论：

```text
机器人VLA研究正在从：
“端到端大模型能不能做动作？”
转向：
“在真实复杂场景里，如何更快、更稳、更可解释、更可控地做决策？”
```

而 CT 介入正好是这个问题最尖锐的版本：
- 实时性要求高，但不是纯高频闭环问题
- 风险高，不允许黑箱“试错”
- 医生必须保留控制权，不可能完全自动化
- 数据小、标注贵、真机实验成本高
- 空间结构强，天然适合图结构与几何建模

所以你的机会不在“做一个医疗版 OpenVLA”。
你的机会在：

> **把 CT 介入任务重新表述成“可解释、高约束、人机协作决策”的问题。**

这反而更有论文辨识度。

## 方向 A：可解释共享控制框架（最优先）⭐⭐⭐⭐⭐

### 主线（树干）

```text
方向A：CT介入机器人中的 Language-Driven Explainable Shared Control
│
│── 背景1：CT介入任务属于高风险、强约束、低容错的人机协作场景
│    ├── 医生擅长高层判断与风险权衡
│    └── 机器人擅长稳定执行、轨迹约束与重复精度
│
│── 背景2：现有工作要么偏自动规划，要么偏图像分割/配准
│    ├── 缺少“高层决策—低层执行”统一框架
│    └── 缺少可解释的人机共享控制机制
│
│── 问题1：黑箱策略难以被医生信任
│    └── 不能解释为什么选这个靶点、这条路径、这个风险规避动作
│
│── 问题2：共享控制缺少语义层接口
│    └── 人类经验、风险偏好、术中临时判断很难显式注入模型
│
│── 方案：Language-Driven Explainable Shared Control Framework
│    ├── 模块1：Task Semantic Graph (TSG) 表达任务、靶点、风险区、器械状态
│    ├── 模块2：E(3) GNN 建模解剖结构与空间关系 ⭐
│    ├── 模块3：Brain–Cerebellum 分层决策
│    ├── 模块4：可解释注意力/注视监督，用于显示“模型看哪里、为什么这么做”
│    └── 模块5：医生 override / approval / veto 共享控制接口
│
│── 验证/实验设计：
│    ├── 仿真环境：CT介入数字孪生 + 穿刺任务场景
│    ├── 数据：术前CT + 合成扰动 + 专家规划样本
│    ├── 指标：靶点误差、路径安全率、医生干预次数、解释一致性
│    └── 对比：纯规划基线 / 纯 imitation baseline / 无解释版本
│
└── 结论/预期结果：
     在不追求完全自动化的前提下，提升安全性、可解释性和协作效率
```

### 为什么这条线最强

1. **最符合 Charles 现有方法论愿景**
   - TSG + E(3) GNN 不是硬塞进去，而是天然契合
   - Brain–Cerebellum 不再只是架构口号，而是能落到“谁负责什么”

2. **最容易讲成完整故事**
   - 医疗场景需要可解释性 → 共享控制是合理中间解 → 图结构和语言接口成为关键

3. **最适合 failure-mode driven 方法**
   - 可以明确分析失败模式：
     - 错选靶点
     - 高风险路径误判
     - 低层执行偏差
     - 人机意图不一致

### 支线（树枝）

```text
│── 支线A1：注视监督是否真的提升医生信任？
│   ├── 学习专家关注区域
│   └── 测试解释热图与专家注视一致性
│
│── 支线A2：共享控制接口怎么定义？
│   ├── 建议式（recommendation）
│   ├── 确认式（approval）
│   └── 接管式（override）
│
│── 支线A3：语言在这里具体做什么？
│   ├── 术前任务描述结构化
│   ├── 风险偏好注入
│   └── 术中状态总结与解释输出
```

### 最小可发表版本（MVP）

- 不需要真做全自动机械臂穿刺
- 先做：
  1. 术前 CT 场景图构建
  2. 靶点/风险区/路径候选生成
  3. 共享控制决策器输出建议
  4. 可解释热图 + 决策理由输出

这一版已经足够像一篇“方法论文”。

## 方向 B：数字孪生驱动的数据闭环与 failure-mode 诊断 ⭐⭐⭐⭐⭐

### 主线（树干）

```text
方向B：Digital Twin-Driven Failure Mode Diagnosis for CT Intervention Learning
│
│── 背景1：医疗机器人数据稀缺，真实失败样本更稀缺
│    ├── 真机采集成本高
│    └── 高风险失败案例难以系统收集
│
│── 背景2：现有VLA/IL方法通常重成功率，轻失败结构分析
│    ├── 很少回答“为什么失败”
│    └── 更少把失败诊断反过来用于数据构建
│
│── 问题1：小数据下模型容易学到脆弱捷径
│    └── 一旦遇到器官位移、成像噪声、针体偏转就崩
│
│── 问题2：数据构建缺乏面向 failure modes 的系统设计
│
│── 方案：数字孪生驱动的 failure-mode 数据闭环
│    ├── 模块1：构造可控扰动仿真器（位移/噪声/遮挡/针偏转）
│    ├── 模块2：定义失败模式 taxonomy ⭐
│    ├── 模块3：对不同 failure mode 定向增广与采样
│    ├── 模块4：训练 decision critic / uncertainty estimator
│    └── 模块5：把诊断结果反哺数据采集策略
│
│── 验证/实验设计：
│    ├── 控制变量生成不同难度等级任务
│    ├── 评估各失败模式上的鲁棒性提升
│    ├── 比较“随机增广” vs “failure-driven增广”
│    └── 分析模型在不同扰动下的解释变化
│
└── 结论/预期结果：
     数据设计本身成为贡献，而不是只把模型越做越复杂
```

### 这条线为什么也很强

这条线几乎是为 Charles 的研究偏好量身定做：
- 强基线优先
- failure-mode driven
- 小而精准改进
- 数据设计当核心贡献

这比“再设计一个新 backbone”更稳，也更容易做出扎实实验。

### 建议定义的 failure taxonomy

```text
CT介入失败模式 taxonomy（初版）
├── F1 目标定位错误
│   ├── 靶点边界模糊
│   └── 多候选病灶混淆
├── F2 路径安全性错误
│   ├── 穿过高风险血管区
│   └── 过近关键器官边界
├── F3 术中偏差未补偿
│   ├── 呼吸运动
│   ├── 组织形变
│   └── 针体挠曲
├── F4 多模态配准错误
│   ├── CT-超声错位
│   └── 术前-术中不一致
├── F5 共享控制失配
│   ├── 机器人建议与医生意图冲突
│   └── 解释无法支撑决策
└── F6 OOD 场景退化
    ├── 罕见解剖结构
    └── 成像质量极差
```

### 这条线的论文优势

- 它不像“纯模型论文”那样内卷
- 更像“方法论 + 实验设计”型工作
- 特别适合先在仿真和公开数据集上做出一版扎实结果

## 方向 C：高层认知规划 + 低层实时执行的分层架构 ⭐⭐⭐⭐

### 主线（树干）

```text
方向C：Brain–Cerebellum Hierarchical Architecture for CT Intervention
│
│── 背景1：CT介入不是单纯抓取或导航，而是长时程高风险决策任务
│    ├── 需要术前规划
│    ├── 需要术中更新
│    └── 需要低层稳定执行
│
│── 背景2：现有VLA常把高层理解和低层动作混在一起
│    └── 导致解释差、调试难、实时性吃紧
│
│── 问题1：高层决策频率低，低层控制频率高，二者耦合不合理
│
│── 问题2：高层模块不应直接输出连续动作，而应输出语义约束或子目标
│
│── 方案：Brain–Cerebellum 分层架构
│    ├── Brain：靶点选择、路径规划、风险评估、共享控制协商
│    ├── Cerebellum：轨迹平滑、偏差补偿、实时控制
│    ├── Safety Layer：硬约束、停机条件、越界检测
│    └── Memory Layer：术前-术中状态对齐
│
│── 验证/实验设计：
│    ├── 比较端到端 vs 分层架构
│    ├── 分析不同刷新频率配置
│    ├── 测试低层扰动补偿能力
│    └── 评估解释粒度与实时性能平衡
│
└── 结论/预期结果：
     在高风险医疗场景中，分层架构优于端到端黑箱策略
```

### 为什么它排第三，不是第一

因为这条线**工程上非常合理**，但**论文辨识度稍弱**。如果只写“分层架构”，很容易被看成系统工程整理；除非你把：
- TSG / E(3) GNN
- 共享控制
- 可解释性
- failure-mode diagnosis

这些一起嵌进去，它才会真正长成一篇强论文。

所以更好的做法是：

> **把方向 C 当系统骨架，方向 A 当主贡献，方向 B 当实验与数据方法支撑。**

这一下子就完整了。

## 6. 未来 2 周最务实的执行路线

### 6.1 第一优先：先做“问题定义 + 系统图”

先别急着继续堆论文。
先产出下面这 4 个东西：

1. **任务定义图**
   - 输入：术前 CT / 术中信息 / 医生指令
   - 输出：靶点建议 / 路径建议 / 风险解释 / 执行约束

2. **Task Semantic Graph 定义**
   - 节点：靶点、器官、血管、神经、针尖、路径点、医生指令、风险事件
   - 边：邻近、包含、穿越风险、时间顺序、因果依赖、操作约束

3. **failure taxonomy 文档**
   - 至少先列 5~6 类失败模式

4. **Brain–Cerebellum 架构图 v1**
   - 每层输入/输出写清楚

这四个东西一出来，整条线就“站住了”。

### 6.2 第二优先：选一个最小实验切口

我建议最小切口是：

> **术前路径建议 + 风险解释**

原因：
- 不需要先把实时闭环做完
- 不需要一上来接真机
- 最容易验证共享控制价值
- 最容易做 ablation

可以定义为：
- 输入：CT体数据 + 候选靶点 + 风险结构分割
- 输出：候选路径排序 + 风险解释文本/热图
- 人类参与：医生接受/拒绝/修改建议

### 6.3 第三优先：实验别一开始就追“大模型胜利”

更稳的实验逻辑是：

```text
基线1：传统启发式路径规划
基线2：纯视觉/纯分割模型
基线3：无图结构的MLP/Transformer版本
基线4：无共享控制/无解释版本
我们的模型：TSG + E(3) GNN + explainable shared control
```

这样对比更能证明“结构化 + 可解释 + 共享控制”的价值。

## 7. 今天最重要的判断：哪些事先别做

### 暂时不要优先做的事情

1. **不要先追求全流程端到端自动穿刺**
   - 太重、太慢、太难讲清楚贡献边界

2. **不要继续无边界扩展论文池**
   - 现在缺的不是“再多 10 篇”，而是“把主线钉死”

3. **不要急着设计全新大模型 backbone**
   - 你的优势不在发明更大的模型，而在结构化任务建模与实验设计

4. **不要把“语言”写得太虚**
   - 语言在这里不是聊天，而是：
